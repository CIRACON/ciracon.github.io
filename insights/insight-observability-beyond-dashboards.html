Observability Beyond Dashboards

Real observability means answering novel questions about system behavior without deploying new instrumentation.

All insights
DevOps Automation 8 min read

Most teams think they have observability because they have dashboards. They have Grafana panels showing CPU usage, request rates, maybe some error counts. They get paged when a metric crosses a threshold. This is monitoring, and it's necessary. But it's not observability.

The difference matters when something goes wrong that you didn't predict. A dashboard tells you about the things you already knew to watch. Observability is the ability to understand what's happening inside your system by examining its outputs — even when you encounter a failure mode you've never seen before.

The three pillars are a starting point, not a destination

You've heard "logs, metrics, traces" repeated enough that it sounds like a complete answer. It's not. These are data types, not capabilities. Having all three doesn't mean you have observability — it means you have the raw materials.

The gap is in how these signals connect. When a user reports that the app is slow, you need to go from that symptom to a root cause. With disconnected tools, the investigation looks like this: check the dashboard for latency spikes, grep through logs for errors around that time, hope a trace ID was logged somewhere, open a different tool to look at traces, try to correlate timestamps between systems.

This works for simple problems. It completely falls apart for complex distributed systems where the issue crosses multiple services.

Structured events over raw logs

The single most impactful change a team can make to their observability is moving from unstructured log lines to structured events. A log line like Processing order 12345 for user 67890 is human-readable but machine-hostile. You can grep for it, but you can't aggregate, filter, or join on it without writing fragile regex patterns.

A structured event for the same operation:

  • service: order-processor
  • event: order.processed
  • order_id: 12345
  • user_id: 67890
  • duration_ms: 47
  • item_count: 3
  • payment_method: credit_card
  • trace_id: abc-123-def

With structured events, you can answer questions you didn't anticipate: "What's the p99 latency for orders with more than 10 items?" "Are credit card orders slower than PayPal orders?" "Which users are experiencing errors most frequently?" You don't need to deploy new instrumentation to answer these — the data is already there.

High-cardinality data is the key

Traditional metrics work well for low-cardinality dimensions: status codes (200, 404, 500), HTTP methods (GET, POST), service names. They break down when you need high-cardinality dimensions: user IDs, request IDs, order IDs, tenant IDs.

But high-cardinality dimensions are exactly what you need during an investigation. "Which specific users are affected?" "Is this happening for a specific tenant?" "Is it correlated with a specific deployment version?" These questions require data dimensions that traditional metrics systems can't handle without combinatorial explosion.

This is where tools like Honeycomb, or structured event stores that support high-cardinality queries, earn their cost. They let you slice and dice by arbitrary dimensions without pre-defining what you'll need. The trade-off is cost — storing and querying high-cardinality data is more expensive than aggregated metrics. But the time saved during incidents often justifies it.

Distributed tracing done right

Tracing is the pillar that teams struggle with most. The instrumentation is more invasive, context propagation breaks in surprising places, and sampling decisions are non-trivial.

A few things that make tracing actually useful:

  • Instrument at the boundaries. Don't trace every function call. Trace service-to-service calls, database queries, cache lookups, and external API calls. These are where latency hides and errors propagate.
  • Add business context to spans. A span that says "HTTP POST /api/orders 200 47ms" is less useful than one that also includes the order ID, user tier, and item count. The business context is what lets you answer "why is this specific request slow?"
  • Head-based sampling is usually wrong. Deciding at the start of a request whether to trace it means you'll sample out the interesting requests (errors, high-latency outliers). Tail-based sampling — making the decision after the request completes — captures the requests that matter. It's harder to implement but worth it.
  • Use OpenTelemetry. Seriously. It's the standard. Vendor-specific instrumentation locks you in and makes it harder to switch tools later. OTel gives you collector-level flexibility to route data wherever you need it.

The cultural shift

Tools don't create observability. Habits do. The teams with the best observability practices share a few traits:

  • Developers instrument their own code. It's not something that happens after deployment — it's part of the development process. If you add a new endpoint, you add instrumentation with it.
  • Incidents start with data, not guesses. The first step in debugging is querying the observability system, not reading code or restarting services.
  • After every incident, the team asks: "Could we have diagnosed this faster with better instrumentation?" If yes, adding that instrumentation is part of the remediation.

Observability is a practice, not a product. You can buy tools that make it easier, but the real work is building the habit of asking questions your system can answer, and extending instrumentation when it can't.
