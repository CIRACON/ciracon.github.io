GitHub and Kubernetes for DevOps Automation

Learn how GitHub integrates with Kubernetes to streamline deployments, automate workflows, and improve DevOps operations.

All insights
DevOps Automation 7 min read

GitHub and Kubernetes fit together well, but most teams make the integration harder than it needs to be. For AI platforms and internal developer platforms, the useful pattern is simple: keep source, CI, and deployment intent in GitHub; let Kubernetes run workloads; and use GitOps or tightly scoped CI deploy jobs to move changes into clusters. The bad pattern is treating GitHub Actions as a long-lived control plane for Kubernetes. It is not. Use it to build, test, sign, and publish artifacts. Let Argo CD or Flux handle steady-state reconciliation inside the cluster.

GitHub to Kubernetes Delivery Flow
1 Commit Code and manifests Developers push application code, Helm charts, or Kustomize overlays into GitHub.
2 Actions Build, scan, publish GitHub Actions builds containers, runs tests, scans dependencies, and publishes signed images.
3 Cluster Sync Argo CD or Flux A GitOps controller in Kubernetes pulls the desired state and reconciles it continuously.

Use GitHub for delivery orchestration, not cluster babysitting

If you are deploying to Kubernetes from GitHub, we recommend one of two models.

  • Preferred for most platform teams: GitHub Actions builds artifacts, then updates a deployment repo or environment branch. Argo CD or Flux applies changes in-cluster.
  • Acceptable for smaller teams: GitHub Actions runs kubectl or helm upgrade directly against the cluster.

For production AI platforms, pick the first model unless you have a very small footprint. GitOps gives you drift detection, better rollback behavior, and a cleaner audit trail. It also separates CI credentials from runtime credentials. That matters once you have multiple clusters, regulated environments, or workloads with GPU nodes and expensive capacity.

Direct deploy from Actions looks simpler on day one. By month six, it usually turns into a pile of brittle workflow YAML, hidden cluster state, and over-privileged service accounts.

How we structure repos and environments

The repo layout matters more than teams expect. If you mix app code, Terraform, Helm charts, and raw Kubernetes YAML in one repo without clear ownership, every deployment becomes a coordination problem.

A practical setup looks like this:

  • Application repo: source code, Dockerfile, tests, and maybe a Helm chart if the app owns it.
  • Platform repo: ingress controllers, cert-manager, external-dns, GPU device plugins, monitoring stack, cluster policies.
  • Environment repo: per-cluster or per-environment manifests, usually with Kustomize overlays or Helm values files.

For AI workloads, keep model-serving config close to the application repo, but keep cluster-wide infra separate. We have seen teams put NVIDIA device plugin config, KServe, app manifests, and team-specific secrets in one repo. That usually ends in accidental blast radius and unclear ownership.

If you use Argo CD, the environment repo becomes the source of truth for what runs in each cluster. A GitHub Action can update an image tag in a Kustomize patch or Helm values file after a successful build.

Authentication from GitHub to cloud and Kubernetes

Do not store long-lived cloud keys or kubeconfigs in GitHub secrets if you can avoid it. Use GitHub OIDC federation with your cloud provider.

  • AWS: GitHub OIDC to assume an IAM role, then push to ECR or update EKS-related resources.
  • GCP: Workload Identity Federation to access GCR or GKE.
  • Azure: Federated credentials with Entra ID for AKS and ACR access.

This is one of the clearest upgrades you can make. It removes static credentials from repo settings and gives you short-lived tokens with auditable trust relationships.

If you still deploy directly from Actions, bind the Kubernetes service account or cloud role to the narrowest scope possible. Most teams get this wrong by giving CI cluster-admin because it is fast. Then a compromised workflow can rewrite ingress, secrets, network policies, and admission controls. For production, that is unacceptable.

A GitHub Actions pipeline that actually works

The pipeline should do a few things well, not many things poorly:

  1. Run unit and integration tests.
  2. Build the container image with docker buildx.
  3. Scan dependencies and images with Trivy or Grype.
  4. Sign the image with cosign.
  5. Push to a registry like ECR, GCR, or GHCR.
  6. Update deployment manifests in a GitOps repo.

For Kubernetes manifests, pick one templating tool and stick with it. We prefer Kustomize for simple environment overlays and Helm when you need reusable packaging or third-party charts. Do not combine raw YAML, Helm, and Kustomize in the same service unless there is a very good reason. That creates debugging debt.

For AI services, also add checks for:

  • GPU resource requests and limits are set correctly.
  • Node selectors and tolerations match your GPU node pools.
  • Model artifact URIs are versioned and immutable.
  • Readiness probes reflect model load time, not just HTTP port availability.

A lot of “Kubernetes problems” in AI platforms are really bad pod specs. The cluster is fine. The deployment contract is wrong.

What we recommend for AI platform workloads

AI workloads stress Kubernetes in ways normal stateless services do not. GitHub automation should reflect that.

For online inference, use separate deployment paths for API code and model artifacts. If every model update requires a full container rebuild, your delivery cycle gets slow and expensive. We prefer immutable model versions in object storage and a deployment manifest that references them explicitly. The app image changes when code changes. The manifest changes when the model version changes.

For batch jobs and training jobs, use Kubernetes-native controllers like Job, CronJob, Ray on Kubernetes, or Kubeflow components where they make sense. Triggering these from GitHub Actions is fine for scheduled automation, but the cluster should own execution and retries. GitHub should not be your job scheduler for long-running ML work.

For secrets, use External Secrets Operator or the Secrets Store CSI Driver with AWS Secrets Manager, GCP Secret Manager, or Azure Key Vault. Avoid encrypted secret blobs committed all over app repos unless your team has strong discipline around key rotation and review.

What usually goes wrong

Most failures are not exotic. They are boring and repeated.

  • CI has too much access. Teams give GitHub Actions broad cluster credentials. One bad workflow change or leaked token becomes a cluster-wide incident.
  • No separation between build and deploy. A successful image build immediately mutates production. There is no promotion step, no environment-specific review, and no clean rollback path.
  • Manifest drift. Engineers hotfix resources with kubectl edit. The cluster diverges from Git. The next sync overwrites the fix, or worse, preserves unknown state.
  • Readiness and liveness probes are wrong. This is common with model servers. Containers restart during model warm-up, then everyone blames Kubernetes.
  • Workflow YAML becomes the platform. Business logic, deployment rules, and environment branching all get buried in GitHub Actions. Nobody can reason about it safely.
  • One cluster for everything. Dev, staging, training, and production inference share the same cluster. Resource contention and noisy neighbors become constant.

The pattern behind these failures is simple: teams optimize for fewer files and fewer repos early, then pay for it with unclear boundaries later.

Lessons learned from running this in production

First, GitOps is worth the extra repo. We resisted this on smaller teams because it looked like process overhead. In practice, it reduced deployment debugging time. When something broke, we could inspect desired state, live state, and sync history in one place.

Second, self-hosted GitHub runners need the same care as any other compute tier. If you run runners inside Kubernetes, isolate them. Use ephemeral runners, lock down egress, and avoid sharing runner pools between trusted and untrusted repos. We have seen teams accidentally create a lateral movement path from CI into internal services because runner pods had broad network access.

Third, policy checks belong before the cluster. Use conftest, OPA, or Kyverno policies in CI to catch bad manifests early. Then enforce the same class of rules in-cluster. Relying only on admission policies slows feedback and frustrates developers.

Fourth, image tags should be immutable. Use commit SHAs, not latest. This sounds obvious, but mutable tags still cause too many rollback and provenance problems.

Concrete recommendations

  • Use GitHub Actions for build, test, scan, sign, and publish.
  • Use Argo CD or Flux for production Kubernetes reconciliation.
  • Use GitHub OIDC instead of static cloud credentials.
  • Split app, platform, and environment ownership into separate repos or clearly separated directories with enforced CODEOWNERS.
  • Use Kustomize for straightforward overlays; use Helm when packaging complexity justifies it.
  • Adopt External Secrets Operator or CSI-based secret integration early.
  • For AI workloads, validate GPU scheduling, probe timing, and model version references in CI.
  • Run policy checks in CI and enforce them again at admission time.

Next steps

If your current setup deploys directly from GitHub into Kubernetes, start by removing static credentials and switching to OIDC. Then separate build from deploy: publish signed images first, and update manifests in Git as a second step. After that, add Argo CD or Flux to one non-production cluster and prove out the workflow before migrating production.

If you already use GitOps, focus next on runner isolation, policy enforcement, and environment repo hygiene. Those three changes prevent most of the operational pain we see in GitHub-to-Kubernetes pipelines.
