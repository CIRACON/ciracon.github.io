Infrastructure as Code: Beyond the Basics

Writing Terraform is the easy part. The hard part is module architecture, state management, and CI/CD at scale.

All insights
Cloud Automation 9 min read

Most teams get the basics of Infrastructure as Code pretty quickly. You write some Terraform, you run terraform apply, your cloud resources appear. Maybe you check it into Git. That part is well-documented and there are tutorials everywhere.

The hard problems show up later, usually when you're managing more than a handful of environments or when more than one team is writing infrastructure code. That's when the architecture decisions you made early on start to either save you or haunt you.

State management is the first real challenge

Terraform state is a serialized representation of what Terraform thinks exists in the real world. When it works, you don't think about it. When it breaks, everything stops.

The beginner mistake is local state files. The next mistake is a single remote state file for everything. Both fail for different reasons: local state can't be shared, and monolithic state files become slow, fragile, and create merge conflicts when multiple people work on infrastructure concurrently.

What works is splitting state by logical boundary. Each major component gets its own state file: networking, compute, databases, monitoring. Each environment (dev, staging, prod) gets its own state. Teams that manage their own infrastructure get their own state. The boundaries should mirror ownership boundaries.

State locking (via DynamoDB for AWS, or blob leases for Azure) is non-negotiable once more than one person can run terraform apply. Without it, two concurrent applies can corrupt your state, and recovery is painful manual work.

Module architecture determines your ceiling

Terraform modules are how you create reusable infrastructure components. Good module architecture makes it easy to spin up new environments and enforce consistency. Bad module architecture creates a tangled web where changing one thing breaks three other things.

A few principles that hold up:

  • Modules should do one thing. A "VPC module" that also creates subnets, NAT gateways, route tables, and security groups is fine — that's one conceptual thing. A module that creates a VPC and also sets up an EKS cluster is doing too much.
  • Keep the interface small. A module with forty input variables is hard to use and hard to maintain. Use sensible defaults. Expose only what actually varies between deployments.
  • Version your modules. When multiple teams consume a shared module, you need versioning so that an update for one team doesn't break another. Git tags work. A private module registry works. What doesn't work is pointing everyone at main.
  • Don't abstract too early. If you've only deployed one environment, you don't know enough about what varies to build a good module. Deploy it inline first. Extract the module after the second or third deployment, when the patterns are clear.

Testing infrastructure code

This is where most teams do nothing, and it shows. Infrastructure code is code. It has bugs. The cost of those bugs — misconfigured security groups, missing backup policies, incorrect DNS records — is often higher than application bugs.

There are a few layers of testing that work in practice:

  • Static analysis. terraform validate, tflint, and checkov (or equivalent policy tools) catch syntax errors, deprecated features, and security misconfigurations before anything touches a cloud provider. These should run in CI on every pull request.
  • Plan review. terraform plan output should be posted as a PR comment. Reviewers should look at what's being created, destroyed, or modified. Automated checks can flag dangerous operations (destroying databases, modifying security groups).
  • Integration tests. For critical modules, use Terratest or a similar framework to actually deploy the infrastructure, verify it works, and tear it down. This is slow and costs money, so reserve it for shared modules that many teams rely on.
  • Drift detection. Run terraform plan on a schedule against production. If someone made a manual change, you want to know about it before it causes a problem.

CI/CD for infrastructure

Running terraform apply from a laptop should be impossible in production. Full stop. All production changes should flow through a pipeline with plan review, approval gates, and audit logging.

The pipeline doesn't need to be complicated. A GitHub Actions workflow that runs plan on pull request, posts the output as a comment, requires approval, and runs apply on merge is enough to start. You can add policy checks, Slack notifications, and automatic drift detection as you grow.

One thing that trips teams up: the pipeline runner needs cloud credentials, and those credentials need to be scoped tightly. Use OIDC federation where possible so you're not storing long-lived secrets. Scope the IAM role to only the permissions the pipeline actually needs. Rotate credentials regularly. The pipeline is a high-value target — treat its security accordingly.

The organizational layer

The hardest IaC problems aren't technical — they're organizational. Who can modify what? How do you enforce standards without blocking teams? How do you handle the team that refuses to use the shared modules?

There's no single right answer, but a few things help: define clear ownership boundaries (this team owns this state), use policy-as-code tools like OPA or Sentinel to enforce guardrails automatically, and make the right thing the easy thing. If your shared modules are harder to use than writing raw Terraform, people will write raw Terraform.

Lessons learned the hard way

The IaC mistake that causes the most damage: mixing application config and infrastructure config in the same repository and pipeline. When a bad Terraform plan can block your application deployments, you've coupled things that should be independent. Separate the lifecycles. Infrastructure changes are slower, riskier, and need different approval workflows than application changes.

State file management is where most teams' IaC journey almost derails. We've seen production outages caused by concurrent state modifications, lost state files, and state that drifted so far from reality that a plan showed 200 resources to destroy. Use remote state with locking from day one. Split state files by blast radius — networking, compute, and application infrastructure should be separate state files so a mistake in one doesn't affect the others.

IaC at scale is an organizational design problem as much as a technical one. The teams that do it well treat their infrastructure codebase with the same rigor — reviews, testing, versioning, documentation — as their application codebase.
