RAG Architecture Patterns That Work

Practical patterns for indexing, retrieval, context management, and evaluation in production RAG systems.

All insights
AI Engineering 10 min read

Retrieval-Augmented Generation has become the default architecture for building AI applications over private data. The idea is simple: instead of fine-tuning a model on your data, you retrieve relevant context at query time and pass it to the model alongside the user's question. Most teams can get a basic RAG prototype working in a day or two.

Getting it to work reliably in production is a different story. The gap between a demo that impresses in a meeting and a system that handles thousands of queries a day without hallucinating or returning irrelevant answers — that gap is where the real engineering happens.

Chunking strategy matters more than you think

The first decision in any RAG system is how to split your documents into chunks for indexing. This is easy to get wrong and hard to debug later, because bad chunking degrades retrieval quality in subtle ways.

Fixed-size chunking (split every 500 tokens) is the common starting point. It's predictable and simple to implement. It also regularly splits mid-paragraph or mid-thought, creating chunks that lack the context needed to be useful when retrieved.

What works better in practice:

  • Semantic chunking. Split at natural boundaries — paragraphs, sections, headers. This preserves meaning within chunks. Libraries like LangChain and LlamaIndex have recursive splitters that handle this reasonably well.
  • Overlapping chunks. Include 10-20% overlap between adjacent chunks. This reduces the chance that a relevant passage gets split across chunk boundaries and lost during retrieval.
  • Hierarchical indexing. For long documents, index both full sections and their sub-sections. Retrieve at the section level for broad questions and at the sub-section level for specific ones. This adds complexity but handles a wider range of query types.

There's no universal ideal chunk size. It depends on your documents, your embedding model's context window, and the kinds of questions users ask. Start with 300-500 tokens with overlap, measure retrieval quality, and adjust.

Retrieval is where most RAG systems fail

The basic retrieval approach — embed the query, find the nearest vectors, return top-k — works for straightforward factual questions. It struggles with anything more nuanced.

Common failure modes:

  • Keyword-sensitive queries. A user asks about "PTO policy" but the document uses "vacation leave." Semantic similarity handles some of this, but not all. Hybrid search (combining vector similarity with BM25 keyword matching) covers more ground.
  • Multi-hop questions. "What's the difference between the Q3 and Q4 pricing tiers?" requires retrieving information about both tiers and comparing them. Single-pass retrieval often gets one but not both.
  • Vague or context-dependent queries. "What did we decide about that?" requires understanding the conversation context to know what "that" refers to. Query rewriting — using the LLM to expand the user's query before retrieval — helps significantly.

Patterns that improve retrieval

Hybrid search combines dense vector retrieval with sparse keyword matching. Run both, merge the results using reciprocal rank fusion, and pass the combined set to the model. This is straightforward to implement with most vector databases and consistently improves recall.

Query decomposition breaks complex questions into sub-questions, retrieves for each independently, and synthesizes the results. For a question like "Compare our US and EU data retention policies," you'd decompose into two queries, retrieve separately, and let the model compare. This takes an extra LLM call but handles multi-hop reasoning much better.

Re-ranking adds a second stage after initial retrieval. Retrieve a larger candidate set (top-20 or top-50), then use a cross-encoder model to re-score and select the most relevant chunks. This is more compute-intensive but catches relevant passages that vector similarity alone misses.

Context window management

You can't just dump all retrieved chunks into the prompt and hope for the best. Context window limits are real, and even within those limits, LLMs perform worse with excessive irrelevant context ("lost in the middle" problem).

Practical approaches:

  • Set a hard token budget for retrieved context (e.g., 3000 tokens) and fill it with the highest-scoring chunks.
  • Deduplicate overlapping chunks before inserting into the prompt.
  • Order chunks by relevance — the most relevant first and last, less relevant in the middle, based on how LLMs attend to context.
  • Include source metadata (document title, section heading) with each chunk so the model can attribute its answers.

Evaluation: how do you know it's working?

This is the part most teams skip, and it's the reason most RAG systems degrade silently over time. Without evaluation, you're flying blind.

Build an evaluation set: 50-100 questions with known correct answers, covering the range of queries your system should handle. Include easy factual questions, harder multi-step questions, and questions where the answer isn't in the corpus (the system should say "I don't know").

Measure at two levels:

  • Retrieval quality. For each test question, are the relevant chunks in the retrieved set? Measure recall@k — what percentage of the time is the right chunk in the top 5 or top 10 results.
  • Answer quality. Is the final answer correct, complete, and grounded in the retrieved context? This can be evaluated manually or with LLM-as-judge approaches (have a model score answers against reference answers). Neither is perfect, but both are better than nothing.

Run this evaluation automatically whenever you change the chunking strategy, embedding model, retrieval parameters, or prompt template. Without it, you won't know if a change improved things or made them worse.

Production concerns

A few things that only matter in production but will ruin your day if you don't think about them:

  • Index freshness. If your documents change, your index needs to be updated. Build an incremental indexing pipeline, not one that re-indexes everything from scratch. Track document hashes to detect changes.
  • Latency. A RAG query chain involves embedding the query, searching the vector database, potentially re-ranking, and then calling the LLM. Each step adds latency. Profile the pipeline end-to-end and optimize the bottleneck (it's usually the LLM call).
  • Cost. LLM API costs scale with token volume. Large contexts mean higher per-query costs. Monitor token usage per query and set alerts for anomalies.
  • Guardrails. The model can still hallucinate even with retrieved context. Add post-processing to check that the answer is grounded in the provided context. Citation extraction helps users verify answers and builds trust.

RAG is a good pattern, but it's not magic. The systems that work well are the ones built with the same rigor as any other production software: tested, monitored, and iterated on based on real usage data.
